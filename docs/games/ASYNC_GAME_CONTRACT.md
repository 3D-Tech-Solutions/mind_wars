# Mind Wars Async Game Contract

This document defines the canonical architecture for every game in Mind Wars.

Use it as the platform-level contract for games played in:

- friend groups
- family groups
- stranger matchmaking
- multiple concurrent Mind Wars involving the same player

The goal is to remove ambiguity. Mind Wars games are not live-session minigames by default. They are asynchronous, offline-first, resumable round challenges delivered through sealed payloads.

## Core Product Model

Mind Wars is an async brain-training platform where a player may participate in many different Mind Wars at the same time.

Each Mind War contains:

- a roster of players
- a minimum of 2 players
- an admin-defined upper player limit
- a round window or deadline model
- a sealed set of game slots selected for that Mind War
- one identical challenge payload per game slot for every player in that Mind War

Each player completes the games in that Mind War at their own pace during the round.

Platform participation rules:

- a player may only be in one `waiting` or planning lobby at a time
- a player may be in multiple active `playing` Mind Wars at once
- the current safety cap for concurrent active Mind Wars is `10`

Solo local play is allowed for practice, but it is not itself a Mind War.

## Non-Negotiable Rules

Every production Mind Wars game must be:

1. Async-first
2. Offline-first
3. Resumable
4. Payload-driven
5. Server-verifiable
6. Safe for concurrent participation across multiple Mind Wars

## Canonical Game Lifecycle

Every game in Mind Wars must follow this lifecycle:

1. The backend locks a Mind War payload for the round.
2. The payload includes one immutable `challengeSet` for each game slot.
3. The client downloads and caches the payload locally.
4. The player starts or resumes progress at any time during the round window.
5. The client stores `progressSnapshot` state locally as the player works.
6. The player completes the game and submits a `finalSubmission`.
7. The server verifies the submission against the immutable payload.
8. The server computes the canonical score and records the result.

## Required Shared Objects

Every game must support these three objects.

### `challengeSet`

Immutable game definition for one game slot in one Mind War.

Required properties:

- `schemaVersion`
- `gameId`
- `challengeKey`
- `canonicalChecksum`
- `difficulty`
- `hintPolicy`
- game-specific deterministic primitives
- any hidden scoring metadata required for server verification

Rules:

- generated by backend during payload lock
- identical for every player in the same Mind War slot
- must not contain client-local timestamps
- must not depend on runtime device randomness

### `progressSnapshot`

Locally persisted resumable game state for one player working on one `challengeSet`.

Required properties:

- `mindWarId`
- `roundNumber`
- `gameId`
- `challengeChecksum`
- `startedAt`
- `lastUpdatedAt`
- `completed`
- game-specific local progress state

Rules:

- stored locally and updated often
- safe to restore after app close, device sleep, or temporary offline periods
- scoped to one player and one Mind War game slot
- never treated as authoritative scoring truth

### `finalSubmission`

Canonical completion package sent to the backend when the player finishes or intentionally submits.

Required properties:

- `challengeChecksum`
- `submissionHash`
- `totalTimeMs`
- `hintsUsed`
- game-specific answer transcript, placements, moves, or selections
- `completedAt` or equivalent completion metadata

Rules:

- created from the sealed payload plus player interaction transcript
- must be deterministic from its contents
- must be replayable or directly verifiable on the server

## Multi-War Concurrency Model

A player may be active in many Mind Wars simultaneously.

The platform should distinguish between:

- planning-lobby participation: maximum `1`
- active Mind War participation: maximum `10` concurrent wars

That means every game implementation must assume:

- the same player may have multiple active `progressSnapshot` records for the same game type
- two active Puzzle Race sessions must not overwrite each other
- two active Path Finder mazes must not share progress state
- local persistence keys must include `mindWarId`, `roundNumber`, `gameId`, and `challengeChecksum`

Never scope saved progress only by `gameId`.

Never assume a small fixed lobby size in the game architecture. A Mind War might involve a small family group or a much larger admin-configured community competition.

## Offline-First Behavior

Mind Wars games must remain playable without connectivity after the payload has been downloaded.

Required behavior:

- cache `challengeSet` locally
- persist `progressSnapshot` locally
- allow full resume during the round
- queue `finalSubmission` if the player finishes while offline
- sync queued submissions when connectivity returns

Rules:

- local progress can be authoritative for resume UX
- server remains authoritative for score, ranking, and final validation

## Design Implications For Game Teams

Game teams should design their games as resumable tasks, not as single-screen live sessions.

That means:

- no required always-online play loop
- no dependence on live turn presence for core puzzle completion
- no hidden local-only randomness in ranked or competitive play
- no score model that trusts the client without verification

A game widget is only one layer of the system.

The full system for every game should include:

1. payload generation
2. payload caching
3. progress persistence
4. resume behavior
5. final submission generation
6. backend verification

## What This Means For Existing Games

Rotation Master:

- sealed prompts
- local resumable progress can be added on top of the current payload model
- final answer transcript must remain server-verifiable

Path Finder:

- sealed maze payload
- local resume should preserve current position, collected objectives, and move transcript
- final route submission must be replayed on the server

Puzzle Race:

- should be built as a sealed image-puzzle system, not a client-randomized puzzle widget
- local resume must preserve piece positions and board state
- final board mapping or placement transcript must be server-verified

All future games:

- must start from this contract instead of retrofitting it later

## Recommended Persistence Key

For local storage, use a composite identity equivalent to:

`mindWarId + roundNumber + gameId + challengeChecksum`

This prevents conflicts across:

- multiple Mind Wars
- repeated appearances of the same game type
- future schema migrations

## Anti-Confusion Summary

Mind Wars is not:

- one live synchronous contest where everyone must play at once
- one active game per player globally
- one local save slot per game type

Mind Wars is:

- many concurrent async contests
- each with sealed game slots
- each playable at the player’s own pace during the round
- each restorable from local progress
- each verifiable by the server at submission time

## Implementation Checklist

Before shipping a game into Mind Wars, confirm:

1. The backend generates and locks an immutable `challengeSet`.
2. The client renders only from that payload.
3. The client persists a scoped `progressSnapshot`.
4. The client can restore progress after app restart.
5. The client builds a canonical `finalSubmission`.
6. The server verifies the submission against the same payload.
7. Local save keys are scoped to a specific Mind War slot, not just the game type.
8. The game remains usable during offline play after payload download.
